* 贡献指南

欢迎为 dscli.el 项目做出贡献！本指南将帮助您了解如何参与项目开发。

* 项目理念

dscli.el 是一个 Emacs 插件，为 [[https://gitcode.com/nanjunjie/dscli][dscli]] 命令行工具提供界面。我们的目标是：

1. 提供简洁、高效的 DeepSeek AI 集成
2. 保持与 Emacs 和 Org mode 生态系统的良好集成
3. 遵循最佳实践和代码质量标准
4. 实践我们推广的工具和方法论

* 开发规范

** 代码风格
- 使用 Emacs Lisp 标准编码风格
- 遵循 lexical binding 模式
- 函数和变量使用有意义的命名
- 添加适当的文档字符串

** 文档规范
- 所有文档使用 Org mode 格式
- 代码示例使用源代码块
- 保持文档与代码同步更新

** 提交规范
- 提交信息使用约定式提交格式
- 每次提交应聚焦于单一功能或修复
- 鼓励使用 dscli 协助编写提交信息

* 提交规范

** 核心原则
我们鼓励使用 dscli 来协助编写提交信息，确保提交信息的规范性和一致性。

** 为什么这样做？
1. 保持项目一致性
2. 实践我们推广的工具
3. 确保提交信息的规范性
4. 展示 AI 辅助开发的实际应用

** 如何操作？
人类开发者编写代码后，可以：
1. 使用 dscli 分析代码变更
2. 让 dscli 协助编写提交信息
3. 人工审核并完善提交信息
4. 执行提交

** 提交信息格式
使用约定式提交格式：
- =feat:= 新功能
- =fix:= 错误修复
- =docs:= 文档更新
- =style:= 代码格式调整
- =refactor:= 代码重构
- =test:= 测试相关
- =chore:= 构建过程或辅助工具

示例：
#+begin_example
feat: 添加模型选择支持

- 新增 dscli-chat-model 自定义变量
- 支持 deepseek-chat 和 deepseek-reasoner 模型
- 更新相关文档和示例
#+end_example

* 开发流程

** 1. 环境设置
1. 克隆仓库
2. 安装 dscli 工具：=go install gitcode.com/nanjunjie/dscli@latest=
3. 配置开发环境

** 2. 创建分支
- 从 =main= 分支创建特性分支
- 分支命名：=feat/xxx=, =fix/xxx=, =docs/xxx= 等

** 3. 开发实现
- 编写代码和测试
- 确保代码质量
- 更新相关文档
- 可以使用 dscli 协助代码编写和审查

** 4. 提交更改
- 暂存更改：=git add .=
- 编写提交信息（鼓励使用 dscli 协助）
- 执行提交：=git commit -m "提交信息"=

** 5. 创建合并请求
- 推送到远程仓库
- 创建合并请求
- 等待代码审查

* 文档要求

** 必须包含的文档
1. 代码文档字符串
2. 用户使用文档（README.org）
3. API 文档（如有）
4. 变更历史（CHANGELOG.org）

** 文档格式
- 使用 Org mode
- 源代码块使用正确的高亮
- 保持一致的标题层级
- 链接使用 Org mode 语法

* 测试要求

** 单元测试
- 关键功能应有单元测试
- 测试覆盖主要逻辑路径
- 测试文件使用 =.el= 扩展名

** 集成测试
- 测试与 dscli 的集成
- 测试用户交互流程
- 确保向后兼容性

* 代码审查

** 审查要点
1. 代码正确性和安全性
2. 是否符合项目规范
3. 是否有适当的测试
4. 文档是否同步更新
5. 提交信息是否规范

** 审查流程
1. 至少需要一名核心贡献者审查
2. 所有审查意见必须被处理
3. 通过审查后由维护者合并

* 发布流程

** 版本号规则
- 主版本号：重大变更，可能不向后兼容
- 次版本号：新功能，向后兼容
- 修订号：错误修复，向后兼容

** 发布步骤
1. 更新 CHANGELOG.org
2. 更新版本号
3. 创建 Git 标签
4. 更新 README.org 中的版本信息
5. 发布公告

* 沟通渠道

** 讨论交流

- 尊重所有贡献者
